# Implementation Plan: Authentication & Role-Based Access Control

**Branch**: `007-auth-rbac` | **Date**: 2026-03-28 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/007-auth-rbac/spec.md`

---

## Summary

Replace the "paste-a-JWT" workaround with a fully functional auth system covering student self-registration (RF-01), login/logout with session management (RF-02), password recovery via email (RF-03), and role-based access control across all 5 platform roles (RF-04).

**Backend**: New `com.klasio.auth` hexagonal module. Real user accounts in a `users` table. JWT access tokens (8h, HMAC-HS256) + DB-backed refresh tokens (7d, rotated on use). Account lockout (5 failed attempts → 15 min). One-time token flows for email verification and password reset. `DevTokenController` deleted.

**Frontend**: Replace the paste-JWT login page with real email+password auth. HttpOnly cookies via Next.js API proxy routes. Role-based routing in `middleware.ts`. New pages: student registration, forgot password, reset password, verify email, and 5 role-specific dashboards.

**Language/Version**: Java 21, TypeScript 5.9

**Testing**: JUnit 5 + Mockito (backend), Testcontainers for integration tests, Jest 29 (frontend)
**Target Platform**: Web (server + client). Next.js App Router middleware runs at the edge.
**Performance Goals**: Login p95 < 200ms (bcrypt factor 12 ≈ 30-40ms; acceptable). Registration p95 < 500ms (email send is async).
**Constraints**: bcrypt factor ≥ 12, TLS 1.2+, server-side session invalidation, no cross-tenant data leakage
**Scale/Scope**: 10k total users at v1.0, 50 tenants

## Architecture Decisions

### AD-01: Lockout state on `users` table, not a separate table

**Problem**: Need to track failed login attempts per user.

**Decision**: Add `failed_login_count` and `locked_until` columns directly on `users`. No separate `login_attempts` table.

**Rationale**: Avoids a JOIN on every login check. The per-user counters don't need historical analysis in v1.0. One atomic `UPDATE users SET failed_login_count = failed_login_count + 1 WHERE id = ?` handles the race condition per spec ("atomic, server-side counter").

---

### AD-02: No token blacklist for access tokens

**Decision**: Access tokens remain valid until natural expiry (8h) even after logout. Only refresh token is revoked.

**Rationale**: The spec explicitly allows this: "existing sessions are not forcibly invalidated in v1.0 but will naturally expire within 8 hours." Adding a Redis blacklist for access tokens is over-engineering for v1.0 scale (10k users).

---

### AD-03: Next.js API proxy for cookie management

**Decision**: Browser → `POST /api/auth/login` (Next.js route handler) → `POST http://localhost:8080/api/v1/auth/login` (Spring Boot) → sets `Set-Cookie` header on the Next.js response.

**Rationale**: `Set-Cookie` on a different origin (Spring Boot port 8080 vs Next.js port 3000) would require `SameSite=None; Secure` which breaks same-site CSRF protection. Routing through Next.js keeps cookies same-origin (port 3000 only).

---

### AD-04: `jose` for JWT verification in Next.js middleware

**Decision**: Use the `jose` npm package (not `jsonwebtoken`) to verify the access token in `middleware.ts`.

**Rationale**: Next.js middleware runs in the Edge Runtime, which does not support Node.js `crypto` module. `jose` is Web Crypto API compatible and explicitly supported in the Edge Runtime. `jsonwebtoken` would throw at runtime in middleware.

---

### AD-05: Role-based route groups

**Decision**: Separate route group prefixes per role: `/superadmin/*`, `/admin/*`, `/manager/*`, `/professor/*`, `/student/*`.

**Rationale**: Clean separation allows `middleware.ts` to enforce RBAC with simple prefix matching. One middleware function handles all 5 roles without per-page guards.

---

### AD-06: Student user account linked to existing student profile via `user_id` FK

**Decision**: Admin-created student profiles (from `005-student-level-assignment`) are not automatically given user accounts. A `user_id` FK is added to `students` (nullable). When a student self-registers, both a `users` row and an update to the existing or new `students` row are created atomically.

**Rationale**: Existing seeded students shouldn't suddenly have broken references. The FK being nullable avoids a data migration for all existing student rows.
