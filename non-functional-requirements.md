# Non-Functional Requirements

Non-functional requirements define the quality attributes of the system and the technical constraints under which it must operate.

All requirements are classified under the following four-level priority scale:

| Level | Name     | Description                                                                                          | Impact if Missing                                          |
|-------|----------|------------------------------------------------------------------------------------------------------|------------------------------------------------------------|
| P0    | Critical | Launch blocker. Without this requirement the system cannot operate safely or coherently.             | System inoperable or insecure.                             |
| P1    | High     | Essential for the core value proposition. Must be present in v1.0 even if temporary workarounds exist. | Key process degraded, requires manual intervention.        |
| P2    | Medium   | Important for user experience or operational efficiency. Can be deferred to v1.1 without compromising the core. | Degraded user experience or additional manual processes.   |
| P3    | Low      | Desirable in the future. Adds differential value but is not necessary in early versions. Later roadmap. | No immediate operational impact.                           |

---

| ID     | Category                    | Description                                                                                                                         | Acceptance Criteria                                                                                                                                                                         | Priority    |
|--------|-----------------------------|-------------------------------------------------------------------------------------------------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|-------------|
| RNF-01 | Security – Authentication   | All communication between client and server must be encrypted. Passwords are not stored in plain text.                             | TLS 1.2+ on all endpoints. Passwords hashed with bcrypt (salt factor ≥ 12). Audit of failed access attempts.                                                                               | P0 – Critical |
| RNF-02 | Security – Authorization    | No user can access data or functionalities outside their role and their tenant.                                                     | Basic penetration tests confirm that no endpoint returns data from another tenant. RBAC implemented on every API endpoint.                                                                  | P0 – Critical |
| RNF-03 | Security – Personal Data    | Sensitive student data (minors, EPS, identity document) must be handled in compliance with data protection regulations (Colombian Law 1581). | Privacy policy published on the platform. Explicit consent at registration. Data of minors subject to additional access restrictions.                                                        | P0 – Critical |
| RNF-04 | Performance                 | The platform must respond with acceptable load times under normal load.                                                             | 95% of HTTP requests respond in < 2 seconds with up to 500 concurrent users. Main page load < 3 seconds on a 4G connection.                                                                 | P1 – High   |
| RNF-05 | Availability                | The platform must be available during the leagues' operating hours.                                                                 | Minimum monthly uptime of 99%. Scheduled maintenance during low-activity hours (overnight). Documented failure recovery plan.                                                                | P1 – High   |
| RNF-06 | Scalability – Multitenant   | The architecture must support multiple tenants without performance degradation.                                                      | Onboarding a new tenant requires no code changes or system restarts. The system supports up to 50 active tenants and 10,000 total students in v1.0 without degradation.                      | P0 – Critical |
| RNF-07 | Usability                   | The interface must be intuitive for non-technical users (students, professors).                                                     | Usability testing with 5 users yields fewer than 3 errors per task in critical flows (registration, payment upload, attendance registration). Responsive design for mobile and desktop.      | P1 – High   |
| RNF-08 | Audit and Traceability      | Every critical action on memberships, payments, and attendance must be recorded.                                                    | The system maintains an immutable audit log: user, action, timestamp, previous value, and new value. Logs are retained for a minimum of 1 year.                                              | P0 – Critical |
| RNF-09 | Notifications               | The system must notify relevant actors about critical events.                                                                       | Transactional emails delivered in < 2 minutes. In-app notifications visible on the dashboard. At minimum: membership expiration (3 days in advance), payment validated/rejected, class cancelled. | P1 – High   |
| RNF-10 | Maintainability             | The code must be modular and documented to facilitate future evolution.                                                             | Unit test coverage ≥ 70% on the business layer. Updated README. API documented with OpenAPI/Swagger.                                                                                        | P2 – Medium |
| RNF-11 | Compatibility               | The platform must work on the most widely used browsers.                                                                            | Compatible with Chrome, Firefox, Edge, and Safari (last 2 major versions). Functional on mobile devices (iOS Safari, Android Chrome).                                                        | P1 – High   |
| RNF-12 | Backup and Recovery         | Data must be protected against loss.                                                                                                | Automated daily database backup with 30-day retention. RTO (Recovery Time Objective) < 4 hours. RPO (Recovery Point Objective) < 24 hours.                                                  | P1 – High   |
