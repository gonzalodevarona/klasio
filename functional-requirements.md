# Functional Requirements

All requirements are classified under the following four-level priority scale:


| Level | Name     | Description                                                                                                     | Impact if Missing                                        |
| ----- | -------- | --------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------- |
| P0    | Critical | Launch blocker. Without this requirement the system cannot operate safely or coherently.                        | System inoperable or insecure.                           |
| P1    | High     | Essential for the core value proposition. Must be present in v1.0 even if temporary workarounds exist.          | Key process degraded, requires manual intervention.      |
| P2    | Medium   | Important for user experience or operational efficiency. Can be deferred to v1.1 without compromising the core. | Degraded user experience or additional manual processes. |
| P3    | Low      | Desirable in the future. Adds differential value but is not necessary in early versions. Later roadmap.         | No immediate operational impact.                         |

## 4.2 Multitenant Management

| RF-06 | Tenant – Program Configuration | Each league can manage its own programs with independent configuration. **Acceptance Criteria:** A program belongs to a single tenant and acts as a lightweight container (name + status). A cross-program plans listing is available for tenant-wide visibility. | P0 – Critical | ✅      |
| RF-38 | Tenant – Manager Role Enablement | The superadmin can enable or disable the **Manager role** for a tenant, so leagues that operate without managers (admin + professors + students only) are not forced to use them. The flag is set at tenant creation (RF-05) — default `true` to preserve current behavior — and can be toggled at any time **only by the superadmin**: the tenant admin (and every other role) cannot change it; the control is absent from all admin-facing UIs and any non-superadmin call is rejected server-side with HTTP 403. Re-enabling is always allowed and unconstrained, and restores the Manager role, navigation, and assignment options. Because admin-and-above already hold every manager permission, disabling never removes operational capability — it only removes the manager layer. The flag is strictly tenant-scoped (no cross-tenant leakage) and every change is written to the immutable audit log (actor = superadmin, previous value, new value, timestamp). | P1 – High | ❌ Not implemented — new requirement specified 2026-06-07. Requires: `tenants.managers_enabled BOOLEAN NOT NULL DEFAULT true` + Flyway migration; superadmin-only endpoint (e.g. `PATCH /tenants/{id}/managers-enabled`) guarded by `@PreAuthorize("hasRole('SUPERADMIN')")`; precondition check before disabling (active managers / manager-assigned plans / `PENDING_MANAGER_ACTIVATION` memberships) → 409 with blocker list; relax mandatory assigned-manager on `ProgramPlan` when disabled (RF-06); conditional hiding of the Manager role in RF-04 role assignment, the RF-20/RF-21 delegation option, and manager nav/dashboard; audit action type `TENANT_MANAGERS_TOGGLED`. Frontend: superadmin toggle in the tenant create/edit form; manager-related UI gated on the tenant flag. |
